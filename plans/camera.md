I want to add a cameras view to the dashboard page.  It should be hidden unless cameras are configured.

I have some rpi 2zerow which have a camera, and should be able to post to a certain endpoint to store the "latest image" for that camera.  For now, i don't intend to save the last N frames, only show last image so there is minimal storage implication.

The settings modal on the dashboard page will allow adding any camera to the cameras view.  The cameras view will just be another tile which can show N differnt cameras.  If no cameras are added, we will not show the cameras view at all.  Please consider the data_log side of this too: /home/nkrueger/dev/data_log.

I think if enabled, on the data_log side, they should periodically post latest image, the rate controllable.  Ideally, the web server / cameras view would be able to request image now, as the rate will probably be long, like 60s or more.  This implies we need a way to send a request to the data_log from the server / cameras page.  It could use the radio cmds, such as 'sample' cmd.  Perhaps since the image relies on WiFi for now we need to have a 2nd optional interface to the data_log piZero2w to listen and accept commands via TCP/IP?  For the TCP/IP communication, i would want it as a 2nd optional communications channel which has the identical cmds, acks, behavior as as the LoRa radio channel.  This channel will not exist for the htcc_ab01 data log impl as they do not use WiFi -- LoRa only.

Communications channel should be an interface and data_log should always interact with the interface, never the LoRa or WiFi implementations directly.  There's a lot of complexity associated with configuring the LoRa that will be greatly simplified for the WiFi channel.  We will likely need interface methods like get/setparam, init, deinit, and so on to satisfy the requirements of the two implementations.  I think for the WiFi channel of data_log, it should implement a simple REST API, and post to the endpoints on the gateway...
